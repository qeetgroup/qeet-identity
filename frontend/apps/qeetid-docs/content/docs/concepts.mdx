Qeet ID is built on five primitives. Almost everything else is a thin wrapper around them.

| Primitive   | Service       | Question it answers                             |
| ----------- | ------------- | ----------------------------------------------- |
| **User**    | `svc-user`    | _Who is this person?_                           |
| **Tenant**  | `svc-tenant`  | _Which organization do they belong to?_         |
| **Role**    | `svc-rbac`    | _What is this user allowed to do?_              |
| **Session** | `svc-session` | _Is this client still authenticated right now?_ |
| **Event**   | `svc-audit`   | _What happened, when, and who did it?_          |

## Users

A **user** in Qeet ID (called a _Qeet ID ID_) is a stable identity that follows a person
across tenants and applications. Users have:

- A canonical **identifier** (`usr_…`) that never changes.
- A **profile** — display name, locale, timezone, picture.
- Verified contact methods — email and phone.
- Free-form **metadata** (JSONB) you control entirely.

```ts
const user = await qee.users.create({
  email: "alex@acme.com",
  profile: { displayName: "Alex Chen", locale: "en-US" },
  metadata: { stripeCustomerId: "cus_xxx" },
});
```

<Callout type="info">
  Users are global. The same Qeet ID ID can be a member of multiple tenants — that's how
  invitations, multi-org switching, and partner workflows work.
</Callout>

## Tenants

A **tenant** (also called an _organization_) is the unit of isolation. Everything else
— roles, sessions, audit logs — lives under a tenant. Tenants control:

- **Branding** (logo, colors, custom domain)
- **Residency** (`us`, `eu`, `apac`)
- **Configuration** (allowed providers, MFA policy, password rules)
- **Admins** (users with privileged rights inside that tenant)

```ts
const tenant = await qee.tenants.create({
  slug: "acme",
  name: "Acme, Inc.",
  residency: "eu",
  branding: { logo: "https://acme.com/logo.svg", primary: "#FF5C2E" },
});
```

## Roles & permissions

Permissions are strings like `"billing:write"` or `"members:invite"`. Roles bundle
permissions. Users (or groups) are **assigned** roles within a specific tenant.

```ts
// Create a role
await qee.roles.create({
  tenant: "acme",
  name: "billing-admin",
  permissions: ["billing:read", "billing:write", "billing:invoice"],
});

// Assign it
await qee.roles.assign({
  tenant: "acme",
  user: "usr_abc",
  role: "billing-admin",
});

// Check it (cached at the edge)
const can = await qee.can({
  user: "usr_abc",
  action: "billing:write",
  resource: "org:acme",
});
```

See [RBAC](/docs/rbac) for the full model, including ABAC attribute policies.

## Sessions

A **session** is a server-validated handle that represents an authenticated client.
Sessions are stateful by default — you can revoke any session from anywhere, instantly,
across your entire fleet.

```ts
// Validate on every request
const session = await qee.sessions.validate(cookie);
if (!session) throw new Unauthorized();

// Revoke from anywhere
await qee.sessions.revoke(sessionId);
```

Sessions can optionally be exchanged for a short-lived JWT for stateless edge use cases.

## Events

Every state change in Qeet ID produces an **event** — `user.signed_in`,
`role.assigned`, `session.revoked`, and so on. Events are:

- Written to an **immutable audit log** with tamper-evident hashes.
- Streamed to your SIEM (Splunk, Datadog, S3, Kafka).
- Available as **webhooks** with retries and at-least-once delivery.

```ts
qee.webhooks.subscribe({
  events: ["user.signed_in", "user.mfa.added"],
  url: "https://api.acme.com/qeetid/webhooks",
});
```

## How they fit

```
                 ┌────────────────┐
                 │     Tenant     │
                 │   (acme)       │
                 └────────┬───────┘
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ▼                 ▼                 ▼
  ┌──────────┐      ┌──────────┐      ┌──────────┐
  │   User   │◄────►│   Role   │      │ Session  │
  │ (usr_…)  │      │ (admin)  │      │ (ses_…)  │
  └──────────┘      └──────────┘      └──────────┘
        │                 │                 │
        └────────► Event log ◄───────────────┘
                  (audit + webhooks)
```
