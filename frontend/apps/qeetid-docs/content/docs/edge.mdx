Every Qeet ID auth call hits the closest edge node. We run in 30+ regions across all
major cloud providers, with sub-50ms p99 sign-in worldwide.

## What runs at the edge

- **Session validation** (`POST /v1/sessions/validate`) — the hot path
- **Permission checks** (`POST /v1/permissions/evaluate`)
- **Tenant configuration lookups**
- **JWT verification** for stateless modes
- **Rate limit enforcement**

## What doesn't run at the edge

- Writes (sign-in, role changes, etc.) — these go to your tenant's region for residency.
- Audit log writes — to a region-pinned, append-only store.
- Webhook delivery — to a queue in the tenant's residency region.

This split is intentional. Reads must be fast and global. Writes must be durable and
residency-compliant.

## Cache layers

Qeet ID uses three cache layers for hot reads:

1. **Edge SRAM** — 30s TTL, key-scoped to (tenant, user, action).
2. **Regional Redis** — 60s TTL, with active invalidation.
3. **Postgres replicas** — strongly consistent fallback.

Cache invalidation is **proactive**: when a role changes, an invalidation event fans
out to all edge nodes within ~200ms.

## Custom domains and edge

If you use a custom auth domain (e.g. `auth.acme.com`), all requests still hit the
nearest edge node. The TLS certificate is provisioned per region; we use a privacy-respecting
sticky-region policy so users typically hit the same node each request.
